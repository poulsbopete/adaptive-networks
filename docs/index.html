The Challenge

Network operations teams need faster triage without risky automation on core routing.

Network teams face

  • High-volume syslog & SNMP telemetry
  • Correlating L2/L3 faults with app impact
  • Slow, manual root-cause analysis
  • Risky auto-remediation on core routing

This demo shows

  • Realistic Cisco-style fault injection
  • Elastic alert → workflow → AI RCA
  • Hybrid correlation with otel-demo K8s & VPC flow
  • Policy-driven auto-fix vs. human approval

Architecture

OTLP simulator → alerts → Kibana Workflow → Agent Builder → remediation poller.

Severity Model

Low severity — automated

MAC flapping and interface error spikes are contained, well-understood failures.

Alert→ AI RCA→ Case→ Auto remediate→ Close

High severity — human in the loop

STP topology changes and BGP session flaps can cause wide outages — require approval.

Alert→ AI RCA→ Case→ waitForInput→ Remediate

Incident Response Workflow

  1. Enrich — ES|QL pull of recent fault logs from logs.otel.adaptive-networks*
  2. Correlate — K8s + VPC flow counts from otel-demo namespace
  3. RCA — Agent Builder conversation (analysis only, no remediation)
  4. Case — Observability case with conversation link & hybrid summary
  5. Branch — severity tag drives auto vs. HITL path
  6. Remediate — agent invokes remediation workflow → queue → poller
  7. Audit — document written to adaptive-networks-incident-audit
Workflow timeout: 24h — paused HITL executions won't stack indefinitely.

Critical guidance

  • Search log text via body.text — not body
  • Vendor mnemonics: MAC flap, STP, BGP, interface errors
  • Always correlate with otel-demo microservice telemetry
  • Remediate only when explicitly instructed
